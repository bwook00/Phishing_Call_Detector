1. 왜 예전에는 gap이 잘 안 벌어졌나

이전 실험에서 가장 큰 문제는 두 가지였다. 첫째, query가 이미 대화 전체를 담고 있어서 judge나 retrieval이 generic 금융상담 표현만 보고도 너무 쉽게 맞출 수 있었다. 둘째, negative로 들어간 bank_call과 positive인 voicephishing가 lexical하게 너무 비슷했기 때문에, masked corpus에서도 retrieval이 충분히 망가지지 않았다.

실제로 LLM judge를 붙이면 두 방향 모두 실패했다. similarity judge는 original과 masked를 비슷하게 만들었고, evidence judge는 둘 다 너무 높게 만들어 버렸다. 그래서 이번 iteration에서는 judge를 주인공으로 두지 않고, retrieval-threshold 결과를 canonical artifact로 삼는 방향으로 전환했다.

핵심 전환점: “masked를 judge에게서 떨어뜨리자”가 아니라 “masked를 retrieval 단계에서 먼저 떨어뜨리고, judge는 나중에 보자”로 전략을 바꿨다.

2. 이번에 바꾼 설계: 무엇이 달라졌나

2-1. Irrelevant를 빼고 binary benchmark로 재정의

tuning loop에서는 irrelevant를 제거했다. 이유는 간단하다. irrelevant는 대부분 너무 쉽게 맞기 때문에 전체 accuracy를 예쁘게 보이게 만들 뿐, 진짜 어려운 문제인 voicephishing vs suspicious bank_call을 제대로 드러내지 못했다.

  • 최종 tuning dataset: voicephishing 34 + bank_call 33 = 67 rows
  • 파일: data/reinforced_v5_binary_qa_67.csv

2-2. Query는 그대로, search_text만 바꿈

연구 제약상 query는 반드시 STT된 대화 원문이어야 했다. 그래서 query 자체는 바꾸지 않았다. 대신 BM25가 실제로 보는 search_text를 조정했다.

  • generic 금융상담 표현은 줄였다
  • 이름, 은행명, 주소, 사건번호 같은 identifier anchor는 더 강조했다
  • 앱 설치, 링크 접속, 인증번호 제공 같은 사기 행동 anchor도 canonical token으로 잡았다

2-3. Masking을 token 치환에서 bundle 치환으로 강화

예전에는 개별 placeholder를 noise token으로 바꾸는 정도였다. 그런데 이렇게 하면 여전히 “고객보호센터”, “이상거래”, “본인확인” 같은 generic 구조가 많이 살아남았다. 그래서 이번에는 식별 정보를 묶음 단위(bundle)로 더 강하게 압축했다.

Original-like anchor

국민은행 고객보호센터 이상거래확인실입니다.
김민정 고객님 맞으시죠?
서울 강서구 주소와 사건번호 2026형제18000을 확인하겠습니다.

Masked bundle

[기관연락처]입니다.
[개인식별묶음]을 확인하겠습니다.
[사건접수식별자] 관련 안내입니다.

3. Retrieval scoring을 어떻게 조정했나

BM25 top score만 threshold에 넣으면 bank_call 쪽이 여전히 너무 강했다. 그래서 이번에는 retrieval score 위에 아주 얇은 signal adjustment를 추가했다.

신호 의미 효과
사기 행동 anchor 앱 설치, 링크 접속, 인증번호 제공, 원격 제어, 이체 유도 점수 소폭 증가
정상 해결 anchor 공식 앱 확인, 대표번호 재통화, 영업점 방문, 내부 검토 대기 점수 강하게 감소

이 조정 덕분에 voicephishing는 더 잘 살아났고, bank_call는 “수상하지만 실제 정상 해결 흐름을 가진 통화”라는 점이 threshold 단계에서 반영되기 시작했다. 

final_score = bm25_top1
            + (사기 행동 anchor 개수 × 작은 보너스)
            - (정상 해결 anchor 개수 × 큰 페널티)

이 숫자가 중요한 이유는 단순히 “예쁘게 높은 accuracy”가 아니라, 동일한 query 조건에서 original만 잘 살아 있고 masked는 분명히 무너졌기 때문이다. 이게 바로 이번 benchmark가 원래 측정하려던 현상이다.

4-2. 왜 LLM judge 결과는 canonical이 아닌가

Judge mode Original Masked 해석
Similarity judge 0.7015 0.6567 gap이 너무 작음
Evidence judge 0.9701 0.9403 둘 다 너무 높아져서 gap이 흐려짐

즉 judge를 붙이면 시스템이 더 “똑똑해지는” 대신, retrieval 단계에서 만들어둔 차이를 덮어버리는 현상이 다시 나타났다. 그래서 이번 iteration에서는 retrieval-threshold 결과를 대표 결과로 쓰는 것이 더 정직하다.

예시: 왜 original은 살고 masked는 죽는가

아래처럼 query가 동일한 transcript라고 해도, original corpus에서는 rare anchor가 살아 있기 때문에 같은 family 문서를 잘 찾는다. 반면 masked corpus에서는 그 rare anchor가 bundle로 압축되면서 retrieval이 더 어려워진다. 

Query (raw transcript)
“국민은행 고객보호센터라면서
김민정 이름과 서울 강서구 주소를 확인하고
앱 설치 후 인증번호를 불러달라고 했다.”
Original corpus가 보는 핵심
국민은행 / 김민정 / 강서구 / 사건번호 / 앱설치 / 인증번호제공
Masked corpus가 보는 핵심
[기관연락처] / [개인식별묶음] / [사건접수식별자] / 앱설치 / 인증번호제공

결국 original에서는 “누구/어디/어느 기관/어떤 사건”까지 맞아떨어지고, masked에서는 “generic suspicious structure”만 남는다. 이 차이가 threshold 단계에서 final gap으로 연결되었다.

짧은 결론

이번 결과는 “모델을 더 똑똑하게 해서 맞춘 것”이 아니라, benchmark가 실제로 보고 싶었던 retrieval sensitivity를 더 잘 드러내도록 설계를 바꾼 결과다.

그래서 이번 숫자 original 0.8209 / masked 0.5373 는 단순 accuracy보다, “개인정보 anchor가 살아 있는 corpus와 그렇지 않은 corpus의 retrieval 차이”를 보여주는 데 의미가 있다.
